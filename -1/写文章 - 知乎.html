3. Focal Loss for Dense Object Detection


focal loss是交叉墒CE loss的改进，CE在0.6还是挺大的
gamma 在0的时候和CE等价， 大于0 对easy样本有限制作用，对困难样本有加权作用。
  1. 增加困难样本的权重，（1-pt)**gamma, pt = p if y = 1 else 1-p
  2. 增加占比较小的类目的权重

Class Imbalance

This imbalance causes two problems:
(1) training is inefficient as most locations are easy nega- tives that contribute no useful learning signal;
(2) en masse, the easy negatives can overwhelm training and lead to de- generate models.
  • Balanced Cross Entropy
  • Model Initialization prior

Robust Estimation

There has been much interest in de- signing robust loss functions (e.g., Huber loss [12]) that re- duce the contribution of outliers by down-weighting the loss of examples with large errors (hard examples). In contrast, rather than addressing outliers, our focal loss is designed to address class imbalance by down-weighting inliers (easy examples) such that their contribution to the total loss is small even if their number is large. In other words, the focal loss performs the opposite role of a robust loss: it focuses training on a sparse set of hard examples.

8. Robust Speech Recognition via Large-Scale Weak Supervision

Data Processing

  1. predict the raw text of transcripts
  2. automated filtering methods
  3. detect and remove machine-generated tran- scripts
  4. ensure that the spoken language matches the language of the transcript according to CLD2.
  5. audio language detector,
  6. We use fuzzy de-duping of transcript texts to reduce the amount of duplication and automatically generated content in the training dataset.
  7. audio files into 30-second segments paired with the subset of the transcript that occurs within that time segment.
  8. including segments where there is no speech
  9. voice activity detection
  10. after training an initial model we aggregated information about its error rate on training and performed manual inspection of these data sources
  11. sorting by a combination of both high error rate and data source size in order to identify and remove low-quality ones efficiently.
  12. To avoid contamination, we perform de-duplication at a tran- script level between the training dataset and the evaluation datasets we thought were at higher risk of overlap, namely TED-LIUM 3 (Hernandez et al., 2018).

model

  1. encoder-decoder Transformer
  2. input: All audio is re-sampled to 16,000 Hz, and an 80-channel log- magnitude Mel spectrogram representation is computed on 25-millisecond windows with a stride of 10 milliseconds.
  3. feature normalization:
  4. scale the input to be between -1 and 1 with approximately zero mean across the pre-training dataset.
  5. encoder
  6. two convolu- tion layers
  7. Sinusoidal position embeddings are then added to the output of the stem after which the encoder Transformer blocks are applied
  8. The transformer uses pre-activation residual blocks
  9. a final layer normalization is applied to the en- coder output.
  10. decoder
  11. learned position embeddings
  12. tied input-output token representations
  13. text tokenizer
  14. for the English- only models
  15. use the same byte-level BPE text tokenizer used in GPT- 2
  16. for the multilingual models
  17. refit the vocabulary (but keep the same size)

Multitask Format

  • tasks
  • voice activ- ity detection
  • speaker diarization
  • inverse text normal-
  • the interface for the model
  • one-to-many mapping to work with a single model,
  • some form of task specification is necessary
  • specify all tasks and conditioning infor- mation as a sequence of input tokens to the decoder
  • audio-conditional language model
  • also train it to condition on the history of text of the transcript in the hope that it will learn to use longer-range text context to resolve ambiguous audio.
  • with some proba- bility we add the transcript text preceding the current audio segment to the decoder’s context.
  • 1. We indicate the beginning of prediction with a <|startoftranscript|> token.
  • 2. predict the language being spoken
  • a. language targets are sourced from the aforementioned VoxLingua107 model.
  • 3. The next token specifies the task
  • 4. specify whether to predict timestamps or not by including a <|notimestamps|> token for that case.
  • timestamp predic- tion
  • predict time relative to the current audio segment
  • add additional tokens to our vocabulary for each of these
  • interleave their prediction with the caption tokens
  • the start time token is predicted before each caption’s text, and the end time token is predicted after.
  • 5. Lastly, we add a <|endoftranscript|> token
  • We only mask out the training loss over the previous context text, and train the model to predict all other tokens.
